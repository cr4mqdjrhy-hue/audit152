# Автоматическая проверка сайта на 152-ФЗ для Tilda

Это минимальный рабочий комплект:

- `tilda-widget.html` - короткий код, который вставляется в Tilda;
- `public/widget.js` - дизайн и логика виджета;
- `public/preview.html` - локальный предпросмотр виджета;
- `server.js` - внешний сервер проверки;
- `package.json` - зависимости сервера;
- `.env.example` - пример настроек.

## 1. Где вставить код в Tilda

1. Откройте нужную страницу в редакторе Tilda.
2. Нажмите `+` для добавления блока.
3. Выберите `Библиотека блоков` -> `Другое`.
4. Выберите блок `T123` / `HTML-код`.
5. Откройте `Контент` блока.
6. Вставьте содержимое файла `tilda-widget.html`.
7. Для локальной проверки оставьте `http://localhost:3000`.
8. Для публичного сайта замените оба адреса `http://localhost:3000` на HTTPS-адрес вашего сервера, например `https://audit.your-site.ru`.
9. Сохраните и опубликуйте страницу.

Важно: в режиме предпросмотра Tilda пользовательский JavaScript может работать не так, как на опубликованной странице. Проверяйте после публикации.

## 2. Нужно ли хостить код отдельно

Да. В Tilda вставляется только виджет. Настоящая автоматическая проверка должна работать на отдельном HTTPS-сервере, потому что ей нужно:

- открывать чужие сайты сервером;
- запускать браузер Playwright;
- видеть cookies, сетевые запросы и скрипты аналитики;
- обходить ссылки на документы.

Tilda не подходит для этой серверной части.

## Локальный предпросмотр

Когда сервер запущен на компьютере, откройте:

```text
http://localhost:3000/preview.html
```

Это страница для проверки дизайна без Tilda.

## 3. Как запустить сервер

На VPS, Render, Railway, Fly.io или другом Node.js-хостинге:

```bash
npm install
npm run install-browsers
npm start
```

Переменные окружения:

```bash
PORT=3000
ALLOWED_ORIGINS=https://your-site.ru,https://your-site.tilda.ws
```

`ALLOWED_ORIGINS` должен содержать домены, с которых будет открыт виджет Tilda.

## 3.1. Как опубликовать через GitHub/GitLab и Render

GitHub/GitLab лучше использовать как хранилище кода. Сам движок нужно запускать на хостинге, который умеет Node.js и Docker.

Рекомендованный простой вариант:

1. Создайте репозиторий на GitHub.
2. Загрузите в него все файлы проекта, кроме `node_modules`, `.env`, `local-server.log`, `local-server.pid`.
3. Откройте Render.
4. Создайте `New` -> `Web Service`.
5. Подключите GitHub-репозиторий.
6. Выберите Docker-деплой. Render возьмет настройки из `Dockerfile`.
7. Добавьте переменную окружения:

```bash
ALLOWED_ORIGINS=https://your-tilda-domain.ru,https://your-project.tilda.ws
```

Для теста можно временно поставить:

```bash
ALLOWED_ORIGINS=*
```

8. Нажмите Deploy.
9. После деплоя откройте адрес вида:

```text
https://your-service.onrender.com
```

10. В Tilda замените локальный адрес в коде:

```html
<div id="audit152-widget"></div>
<script
  src="https://your-service.onrender.com/widget.js"
  data-api-url="https://your-service.onrender.com/audit152"
></script>
```

11. Опубликуйте страницу Tilda.

Если используете GitLab, последовательность такая же: репозиторий хранится в GitLab, а Render/Railway/Fly.io подключается к нему или получает код через импорт.

## 4. Что проверяет MVP

Сервис проверяет открытые признаки:

- открывается ли сайт;
- есть ли ссылка на политику обработки персональных данных;
- похож ли документ политики на полный;
- есть ли поля, похожие на сбор персональных данных;
- есть ли чекбокс согласия;
- не отмечен ли чекбокс заранее;
- есть ли cookie-баннер;
- есть ли кнопка отказа;
- запускаются ли Яндекс.Метрика, Google Analytics, VK/Mail.ru/myTarget или Meta Pixel до согласия.
- верхнюю ориентировочную оценку штрафа РКН по найденным рискам для юридического лица.

## 5. Важное ограничение

Это не юридическое заключение. Автоматическая проверка не видит внутренние документы компании, договоры с обработчиками, реальные сроки хранения данных, место хранения баз и порядок реагирования на инциденты. Суммы штрафов указаны как верхняя оценка риска по ст. 13.11 КоАП РФ и не являются прогнозом постановления РКН или суда.
